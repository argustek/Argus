### 2. V2 Bridge (`internal/chat/bridge.go`)
- Connects ArgusCore ↔ MessageBus ↔ Frontend

- `emitStatus()` → phase:role:status format for role lights
- Status flow: `Core.emitStatus()` → `Bridge.onCoreMessage("status",...)` → `app.go callback` → `emitToFrontend("role-status",...)` → `frontend EventsOn('role-status')`

### 3. emitToFrontend — Single Pipe (`app.go`)
- All events through MessageBus ONLY (no dual `EventsEmit` fallback)
- Payload fully merged into msgBus data param (id/role/content/timestamp all included)
- Checksum auto-generated per message

### 4. MessageBus SetContext (`message_bus.go`)
- Added `func (mb *MessageBus) SetContext(ctx context.Context)` method
- Called during Bridge init: `GetMessageBus().SetContext(a.ctx)`
- Fixes `mb.ctx == nil` → events not emitted issue

### 5. shouldTrack — Narrow Scope (`message_bus.go`)
- Only `PathCoreOutput` enters pending queue for ACK
- All other paths (system, status, stream, exec, user_input) are no-track
- Fixes false "消息丢失" alarms for system events

### 6. Frontend Timestamp Fix (`frontend/src/App.vue`)
- Changed 5× `new Date().toISOString()` → `Math.floor(Date.now() / 1000)`
- Root cause: ISO string → parseInt → 2026 → Jan 1 1970 → "1/1 08:33"
- Now uses Unix seconds, consistent with backend

### 7. Frontend role-status Listener (`frontend/src/App.vue`)
- Added `EventsOn('role-status', ...)` — parses `phase:X|role:Y|status:Z`
- Updates `aiStatus.pmStatus/seStatus/apStatus` real-time

## Current Status / Pending 🔴

### Frontend Blank Issue
**Symptom:** Frontend shows nothing — no messages, no indicators.
**Last working state:** Dual-path (msgBus + EventsEmit fallback) — messages displayed correctly.
**Broken by:** Single-pipe refactor — `emitToFrontend` now only calls `msgBus.Send()` + `return`.
**Suspected root cause:** MessageBus ctx set correctly but `EventsEmit(mb.ctx, eventName, enrichedData)` may have format mismatch with frontend event handlers. Frontend expects raw payload format (`{id, role, content, timestamp}`), but msgBus wraps everything in `{_msgId, _role, _path, ..., id, role, content}` — the extra keys may break frontend event processing.

**Next step:** Check if frontend `new-message` handler works with msgBus-enriched data format. May need to either:
a) Strip msgBus metadata before EventsEmit, or
b) Keep dual-path: msgBus for tracking, EventsEmit for display (back to working state)
