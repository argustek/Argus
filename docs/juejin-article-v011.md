# 一个人 + AI，15天开发了一款多智能体编程助手 Argus

> 从电气工程师到独立开发者，我用AI在半个月里捣鼓出了一个能自动写代码、自己审核、还能自我修复的桌面端AI编程助手。

## 一、我被 AI 编程工具"卡"怕了

过去一年，我深度使用过 Cursor、OpenCode、Claude Code 等主流 AI 编程工具。它们确实很强大，但总有几个让我抓狂的问题：

- **说干完了，其实没干**：AI 回复"任务完成"，然后就没动静了，文件没生成，命令没执行。
- **干到一半卡死**：AI 陷入死循环或忘记上下文，你得反复点"继续"。
- **环境不认**：每次告诉它"Go 装在 D:\tools\go"，下次对话它又忘了。
- **毫无人情味**：它不知道现在是凌晨三点，也不知道你连干了八小时。

我实在受不了了，决定自己造一个。

## 二、Argus 的核心理念：让 AI 像真正的工程团队一样协作

我不是软件科班出身，是个搞电气和测试的工程师，平时用 LabVIEW 图形编程。所以我脑子里天然就有一个想法：**一个人搞不定的事，那就组个团队**。

Argus 不是"一个 AI"在战斗，而是**四个角色**分工协作：

- 🎯 **PM（项目经理）**：理解你的需求，拆解任务，分发给 SE，最后审核结果。
- 💻 **SE（软件工程师）**：根据 PM 的安排写代码、改文件、执行命令。
- 🔍 **AP（独立审批人）**：在 SE 完成后，不受 PM 影响，独立运行编译/测试，如果通不过就一票否决，打回重做。
- 📊 **C（后台监控）**：默默盯着 PM 和 SE 的健康状态，发现卡死就自动重启，还会定时自动提交 Git。

```
👤 用户（你）
   ↓ 自然语言输入
🎯 PM（项目经理）— 理解需求 → 拆解任务 → 分配给SE
   ↓ 任务分配 + 质量控制
💻 SE（软件工程师）— 写代码 / 改文件 / 执行命令
   ↓ 任务完成 → PM审核通过
🔍 AP（审批人）— 独立代码审查 + 编译测试（一票否决权）
   ↑
📊 C（监控者）— 健康检查 / Git自动提交 / 卡死检测（只报告不决策）
```

这样一来，**整个开发过程变成了一个可以观察、可追溯、能自动恢复的工程闭环**。用户只需要在聊天窗口里说"帮我写一个 Hello World"，PM 拆任务，SE 写文件，AP 审批，C 在背后兜底。

## 三、一个人 + AI，15天从零到发布

我 5 月 1 日开始写第一行代码，5 月 16 日发布了 v0.1.0，现在已经是 v0.1.1 了。**项目约 20 万行代码**，这背后全靠 AI 帮我生成大量样板代码和调试。

技术栈：

- **Go 1.22+**：后端核心，性能稳定，并发友好
- **Wails v2**：用 Go + Web 技术打包桌面应用，最终 exe 只有 ~15MB，启动秒开
- **Vue 3 + TypeScript**：前端界面，聊天、编辑器、终端、文件树
- **Monaco Editor**：VS Code 同款编辑器
- **xterm.js**：内嵌终端

整个项目结构清晰，模块化拆分，已经开源在 GitHub 上。

## 四、目前能做到什么（v0.1.1）

### ✅ 核心功能

- ✅ **自然语言写代码**：跟 PM 聊天，自动拆解任务，SE 生成文件并执行命令
- ✅ **多 AI 独立审批**：AP 用自己的模型重新编译/测试，防止假完成（一票否决权）
- ✅ **SSE 流式输出**：逐字显示 AI 的思考过程，不用干等
- ✅ **任务记忆与崩溃恢复**：不小心关了软件，重启后能接着干
- ✅ **环境记忆**：第一次告诉它 Go 的路径，它就记住了
- ✅ **钉钉集成**：可以用手机钉钉远程触发任务
- ✅ **完整的桌面 GUI**：文件树、编辑器、终端、Git 面板、暗色主题
- ✅ **四角色协作架构**：PM→SE→PM Review→AP Approval 完整流水线
- ✅ **C 监控系统**：30 秒健康检查、Git 变更检测、超时兜底机制
- ✅ **消息去重机制**：前后端双重过滤，确保消息不重复显示

### 🔧 技术亮点

| 特性 | 说明 |
|------|------|
| **四角色架构** | PM/SE/AP/C 各司其职，模拟真实团队 |
| **AP 一票否决** | 不受 PM 影响，独立运行编译测试 |
| **SSE 流式推送** | ai-stream-chunk / pm_message / ap_message 多通道事件驱动 |
| **消息 ID 追踪** | 每条消息唯一 ID，前后端一一对应 |
| **审计日志系统** | 自动记录后端送水时间线，可对比前端收水情况 |
| **状态机管理** | idle → running → done → approved 四态流转 |
| **限流熔断保护** | 防止 API 过载和级联故障 |

## 五、不足与后续计划

毕竟是 MVP 版本，还有不少已知问题：

- 目前只支持 Windows（macOS/Linux 后续会补）
- 复杂任务偶尔会死锁（正在修，已实现 G49-G57 多轮修复）
- 测试覆盖率还很低
- 需要手动配置 API Key

但核心闭环已经能跑通，我每天都在用自己写的工具写代码。

**v0.2.0 计划**：本地模型（Ollama）、VS Code 插件、更完善的测试、插件 SDK。
