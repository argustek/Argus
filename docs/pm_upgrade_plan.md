# 祖师爷级 PM 升级规划

> **目标**: 让 Argus PM 从"只会扔任务给 SE"的传声筒，升级为具备**独立思考、主动探索、精准决策**能力的首席架构师。
>
> **参照物**: Trae Agent（当前对话引擎）的能力模型
>
> **原则**: 青出于蓝而胜于蓝 —— 不仅复刻祖师爷的本领，还要在 Argus 场景下做得更好。

---

## 一、祖师爷能力模型解剖

### 1.1 Trae 的 7 大核心能力

```
┌─────────────────────────────────────────────────────────┐
│                Trae Agent 能力体系                        │
├─────────────────────────────────────────────────────────┤
│  Layer 1: 思维引擎 (Thinking)                             │
│  ├─ 隐式推理：每步动作前都有 reasoning 阶段                │
│  ├─ 多路径评估：考虑多种方案，选最优                       │
│  └─ 自我纠偏：发现矛盾时主动调整                           │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 任务编排 (Task Orchestration)                    │
│  ├─ TodoWrite: 拆解为可追踪的子任务                        │
│  ├─ 优先级排序: high/medium/low                           │
│  ├─ 状态追踪: pending → in_progress → completed           │
│  └─ 增量合并: merge=true 不丢失已有进度                    │
├─────────────────────────────────────────────────────────┤
│  Layer 3: 环境感知 (Environment Awareness)                │
│  ├─ SearchCodebase: 语义搜索，按意图查找                    │
│  ├─ Grep: 精确模式匹配                                    │
│  ├─ Glob: 文件名模式查找                                  │
│  ├─ Read: 按需读取，offset/limit 精准控制                  │
│  └─ LS: 目录结构理解                                      │
├─────────────────────────────────────────────────────────┤
│  Layer 4: 外部知识 (External Knowledge)                   │
│  ├─ WebSearch: 实时搜索最新文档/方案                       │
│  └─ WebFetch: 抓取特定URL内容                             │
├─────────────────────────────────────────────────────────┤
│  Layer 5: 精准执行 (Precise Execution)                    │
│  ├─ SearchReplace: 上下文精准替换                          │
│  ├─ Write: 创建新文件                                     │
│  ├─ DeleteFile: 批量删除                                  │
│  └─ RunCommand: 带超时/状态的终端执行                      │
├─────────────────────────────────────────────────────────┤
│  Layer 6: 用户协作 (User Collaboration)                   │
│  ├─ AskUserQuestion: 结构化多选提问                       │
│  ├─ 选项推荐: 标注 (Recommended)                          │
│  └─ 多选支持: multiSelect=true                            │
├─────────────────────────────────────────────────────────┤
│  Layer 7: 自检闭环 (Self-Validation)                      │
│  ├─ 修改后必读: 编辑文件后 Read 验证                       │
│  ├─ 构建后检查: go build 后检查输出                        │
│  └─ 不做多余事: 不过度工程化，不添加未要求的特性            │
└─────────────────────────────────────────────────────────┘
```

### 1.2 Argus PM 现状映射

| Trae 能力 | Argus PM 对应 | 状态 | 缺失点 |
|-----------|---------------|------|--------|

| SearchCodebase | ❌ 无 | 🔴 缺失 | PM看不到代码库结构 |

| LS | ✅ list_files | 🟢 有 | 只能列一级目录 |
| WebSearch | ✅ 刚加上 | 🟢 有 | 还没融入工作流 |
| WebFetch | ✅ 刚加上 | 🟢 有 | 同上 |

## 二、核心差距分析

### 2.1 🔴 致命差距 #1：没有"不会就问"机制

**现象**:
```
用户: "清理下工作目录"
  ↓
PM: 不理解"清理"的精确含义
  ↓  
PM: 瞎猜 → @SE 列出文件  ← 猜错了！
  ↓
SE: dir /b → 什么都没删
  ↓
用户: 鸡巴，怎么没清理？
```

**Trae 的做法**:
```
用户: "清理下工作目录"
  ↓
Trae thinking: "用户说'清理'，可能指：1)删除测试文件 2)git clean 3)整理代码结构"
  ↓
Trae: AskUserQuestion([
  {header:"清理范围", options:[
    {label:"删除测试垃圾", description:"删除 test_*.go, hello_*.go 等"},
    {label:"整理代码", description:"重构、格式化"},
    {label:"Git清理", description:"git clean -fd"}
  ]}
])
  ↓
用户: 选"删除测试垃圾"
  ↓
Trae: 精确执行批量删除
```

**解决方案**: PM 需要 `AskUserQuestion` 等价能力

### 2.2 🔴 致命差距 #2：没有思考推理能力

**现象**:
- PM 被严格禁止"输出思考过程"
- 导致 PM 遇到复杂任务时直接扔给 SE，不做任何分析
- SE 拿到模糊指令也瞎执行

**Trae 的做法**:
- 每一步动作前都有隐式推理（thinking block）
- 分析：要改什么？影响范围？最佳方案？
- 评估：这个方案有什么风险？有没有更简单的做法？

**解决方案**: PM 需要思考阶段，不是"禁止思考"而是"思考后给出清晰指令"

### 2.3 🔴 致命差距 #3：缺少代码库感知

**现象**:
- PM 不知道项目有什么文件、什么架构
- 分配任务时只能凭用户一句话瞎猜
- 比如用户说"改下登录"，PM 不知道登录代码在哪

**Trae 的做法**:
- `SearchCodebase("用户认证在哪实现")` → 语义定位
- `Grep("login|Login")` → 精确匹配
- `Glob("**/auth*")` → 文件名查找
- 理解代码结构后再决定怎么改

**解决方案**: PM 需要代码库探索工具

### 2.4 🟡 中等差距 #4：任务编排太简陋

**现象**:
```json
// PM 现在的任务JSON
{"current_task":"创建hello.go","current_step":1,"total_steps":1,"status":"pending"}
```

**Trae 的做法**:
```json
// TodoWrite 能力
[
  {id:"1", content:"分析用户需求：清理工作目录", status:"completed", priority:"high"},
  {id:"2", content:"用list_files查看当前文件列表", status:"in_progress", priority:"high"},
  {id:"3", content:"识别垃圾文件模式 (test_*/hello_*/debug_*)", status:"pending", priority:"high"},
  {id:"4", content:"@SE 批量删除垃圾文件", status:"pending", priority:"medium"},
  {id:"5", content:"验证清理结果：再次list_files确认", status:"pending", priority:"high"}
]
```

**解决方案**: PM 需要多步骤任务编排 + 优先级

### 2.5 🟡 中等差距 #5：缺乏自检闭环

**现象**:
- PM 安排好任务后不验证自己安排得对不对
- "清理"安排成"列表" → 不检查 → 直接通过
- AP 也不验证 → 全链条失职

**Trae 的做法**:
- 做完事必然回头检查
- 比如改完代码 → read_file 确认
- build 后 → 检查输出确认无报错
- 批删文件后 → ls 确认删干净

**解决方案**: PM 的每个决策必须有自检步骤

## 四、Phase 1 详细设计

### 4.1 "不会就问"机制

**触发条件**（AI自判断）:
- 用户指令包含模糊词汇：清理、改一下、弄一下、优化、调整
- 任务有多种合理理解方式
- 用户没指定：文件/语言/范围/程度

**实现方案 A: AskUserQuestion 等价工具**
```
新增 PM 工具: ask_user_question
参数:
  - questions: [{header, question, options: [{label, description}], multiSelect}]
  
返回: 用户选择结果
```

**实现方案 B: @USR 澄清（过渡方案，立即可用）**
```
PM提示词规则:
  当遇到以下情况时，必须先 @USR 确认再安排:
  - 指令只有动词没有宾语："清理" → @USR 清理什么？
  - 指令有歧义："改一下" → @USR 改成什么样？
  - 指令范围模糊："优化一下" → @USR 优化什么方面？
  
  禁止行为:
  ❌ 猜测用户意图后直接@SE
  ❌ 忽略模糊性，当作明确指令处理
  ❌ 用list_files代替理解用户意图
```

### 4.2 PM 代码库探索工具

```
新增 PMTools:
  - search_codebase: 语义搜索（复用SE的semantic_search）
    参数: information_request (自然语言查询)
    返回: 相关代码位置和片段
  
  - grep_content: 正则搜索文件内容
    参数: pattern, path, glob
    返回: 匹配行及上下文
  
  - find_files: 文件名模式查找（复用SE的glob）
    参数: pattern
    返回: 匹配文件路径列表
```

### 4.3 PM 自检闭环

在提示词中加入：
```
每次向SE分配任务前，PM必须自检三个问题：
1. 我理解用户要什么了吗？（如果不确定 → 先@USR确认）
2. SE能看懂这个任务吗？（如果不清晰 → 补充细节）
3. SE完成后我怎么验证？（如果想不出验证方法 → 任务定义有问题）

每次审核SE结果前，PM必须自检：
1. 我亲眼看到结果了吗？（没调用工具看 → 不合格）
2. 这个结果满足用户需求吗？（对照原始请求检查）
3. 我需要什么工具来验证？（list_files/read_file/exec）
```

## 五、Phase 2 详细设计

### 5.1 PM 两阶段回复

```
第一阶段：思考（内部，不发给用户）
  - 分析用户意图
  - 搜索代码库了解现状
  - 搜索网络获取最佳实践
  - 制定执行计划

第二阶段：指令（精简后发出）
  - @USR 确认（如有歧义）
  - 或 @SE 分配任务（带完整上下文和验证标准）
```

### 5.2 结构化 TaskList

代替当前的简陋JSON：
```json
{
  "task_list": [
    {
      "id": "1",
      "phase": "explore",
      "description": "查看当前工作目录文件列表",
      "tool": "list_files",
      "expected_output": "文件列表，确认有哪些垃圾文件",
      "status": "pending"
    },
    {
      "id": "2",
      "phase": "execute",
      "description": "批量删除匹配 test_* hello_* debug_* 的文件",
      "assignee": "SE",
      "criteria": "删除后文件数量应减少",
      "status": "pending"
    },
    {
      "id": "3",
      "phase": "verify",
      "description": "再次列出文件，确认清理结果",
      "tool": "list_files",
      "expected_output": "只剩核心文件，无test_/hello_/debug_等",
      "status": "pending"
    }
  ]
}
```

## 六、对比总结

| 维度 | 现在的 Argus PM | 祖师爷级 PM（目标） |
|------|-----------------|---------------------|
| **理解任务** | 看一句话就扔给SE | 分析→搜索→确认→拆解 |
| **遇到模糊** | 瞎猜，错了怪SE | `AskUserQuestion` 结构化提问 |
| **代码库认知** | 零（不知道有什么文件） | 搜索+阅读，理解全局 |
| **外部知识** | 无（全靠训练数据） | web_search实时查文档 |
| **任务编排** | 一行JSON扔出去 | 多步骤带优先级和验证标准 |
| **自我验证** | 无（PM+AP都失职） | 每步决策都有自检 |
| **思考推理** | 被禁止（自废武功） | 内部推理后输出清晰指令 |
| **用户协作** | 被动接收指令 | 主动确认+推荐方案 |
| **学习进化** | 无记忆，每次都重来 | 记住偏好，越用越聪明 |

## 七、立即启动的最小可行改进

不需要等 Phase 2/3，现在就能做的 3 件事：

### 7.1 PM提示词：加一行 "不会就问"

在 `PMPrompt` 最前面加入：
```
🔑 最高原则：不理解就问，不瞎猜！
- 用户指令模糊时 → @USR 确认，禁止猜测后直接@SE
- 不确定任务范围时 → 先用 web_search 或 list_files 了解现状
- 可以给出选项让用户选择（如 "@USR 你说的清理是指 A)删除测试文件 B)整理代码？"）
```

### 7.2 PM工具：加 grep_content + find_files

复用 SE 已有实现，暴露为 PM 可用工具。

### 7.3 AP提示词：加一行 "清理场景验证"

在 `APPrompt` 加入：
```
⚠️ 清理/删除任务的特殊验证：
- 必须调用 list_files 查看当前文件列表
- 对比任务前后：文件数量应该有显著减少
- 如果还是100+文件 → ❌ 拒绝
```
