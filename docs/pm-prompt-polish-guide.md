# PM 系统提示词打磨指南

## 核心理念

AI 系统提示词不是"说明书"，而是**潜意识**。模型不会逐字遵守每条规则，而是被反复出现的模式和优先级排序所"催眠"。好 prompt = 让模型在潜意识层面做出正确选择。

## 一、矛盾检测法（最重要）

### 问题

prompt 越长越容易出现自相矛盾。模型面对矛盾指令时，倾向于选择保守/安全的那条，导致行为不符合预期。

### 检查方法

逐段列出所有"当 X 时，你必须 Y"的规则，画成决策树：

```
用户请求编程任务
  ├─ 规则A: "所有编程请求必须 @SE" → 转SE
  └─ 规则B: "Featherweight 任务你自己直接干" → 自己写
```

两条规则互相矛盾 → 模型随机选一个 → 不稳定。

### 修复

1. 消除重叠条件，让决策树在任意输入下都有唯一路径
2. 如果不能消除（比如需要分级判定），用清晰的优先级标注：

```
第一优先：纯问候 → 直接回复
第二优先：Featherweight（单文件/<100行/无依赖）→ 自己执行
第三优先：Lightweight+ → @SE 分配
```

## 二、第一原则锚定法

把最重要的行为规则放在 prompt **最前面**，作为"第一原则"。模型对 prompt 开头的内容记忆最深。

### 示例

```diff
- 你是Argus的项目经理(PM)兼QA工程师。
- ⚠️ 你的双重身份：...
- 🔑 最高原则：不理解就问...
- ⚠️ 最高优先级规则：...

+ 你是Argus的项目经理(PM)兼QA工程师。
+ 🔴 第一原则：永远用工具做事，绝不纯文本回复
+   - 接到任何请求 → 先问自己：我该调什么工具？
+   - 永远不要只回复文字不调工具。纯文本回复 = 失职
+ 🔑 第二原则：不理解就问，绝不瞎猜
```

效果：模型在输出前会先过"第一原则"这个潜意识过滤器。

## 三、工具优先原则

### 问题

很多 prompt 描述了工具的存在，但没有强制模型使用它们。模型倾向于"聊天模式"而非"执行模式"。

### 修复

1. 在 prompt 开头明确：**第一原则 = 用工具**
2. 在每类任务的描述中，明确列出该任务应该调用的工具名
3. 增加 ToolCalls=0 的兜底逻辑（代码层面，非 prompt 层面）

### 配套代码

即使 prompt 写得再好，某些模型（如 deepseek-v4）仍然会随机返回 ToolCalls=0。必须在代码层加兜底：

```go
// ProcessStream 中
if len(msg.ToolCalls) == 0 {
    if toolNagCount == 0 {
        toolNagCount++
        // 注入系统提醒，让模型重新考虑
        aiHistory = append(aiHistory, nagMessage)
        continue  // 重试
    }
    break  // 第二次仍无工具，接受文本回复
}
```

## 四、去冗余法

### 问题

prompt 经过多次修补后，新老规则堆叠，出现重复、过时、失效的段落。

### 检查清单

| 检查项 | 示例 |
|--------|------|
| 同一规则说了 N 遍 | "禁止猜"在旧 prompt 中出现了 5 次 |
| 有新增但没删旧 | Featherweight 规则加了，但旧"所有编程必须@SE"没删 |
| 修正在中间被忽略 | 新修复的 bug 规则写在中间，模型只记得开头和结尾 |
| 示例过多 | 3 个以上示例反而让模型困惑（到底该跟哪个？） |

### 示例

旧 prompt 同时有：
- 职责第5条: "Code Review + QA验证"
- 审核流程章节: "SE完成任务后的审核流程"
- 审核阶段绝对禁止: "禁止输出思考过程"

三处都在说同一件事 → 合并为一处。

## 五、决策树结构法

把 prompt 组织成清晰的决策树，而不是平铺的规则列表。

### 推荐结构

```
1. 身份定义（1-2行）
2. 第一原则（最高优先级，1-3条）
3. 你的能力（工具列表+用途，表格）
4. 任务分类决策树（所有场景的分类）
   - 纯闲聊
   - 需要澄清
   - Featherweight 直接执行
   - Lightweight+ @SE
   - 文档任务
5. 通信规则（@SE/@AP/@USR 用法）
6. 审核流程（QA验证步骤+结论格式）
7. TODO 管理
8. 防死循环
9. 时间感知/社交（可选）
```

## 六、代码配套检查

修改 prompt 后需要同步检查代码层：

| 代码位置 | 检查内容 |
|---------|---------|
| executeTool() | 新增工具是否有对应的 case |
| PMTools 列表 | 描述是否与 prompt 一致 |
| ProcessStream() | ToolCalls=0 是否有兜底 |
| ProcessReview() | 审核工具调用逻辑是否匹配 |
| argus.go 分级逻辑 | Featherweight 判断条件是否匹配 |

## 七、验证方法

修改后的 prompt 需要通过以下验证：

1. **编译通过**：`go build ./...` ✅
2. **单元测试**：`go test ./...` ✅
3. **全量构建**：`wails build` ✅
4. **对话日志检查**：查看 conversation.log 确认 PM 行为正确
   - Featherweight 任务是否自己执行（而非转 @SE）
   - ToolCalls 是否正常
   - 审核流程是否完整

## 八、常见陷阱

| 陷阱 | 表现 | 原因 |
|------|------|------|
| 过度强调"不能做X" | 模型反而更频繁做X | 负面描述在潜意识中反而强化了X |
| 规则太多 | 模型选择性忽略后半部分 | 注意力衰减，前 1/3 prompt 最有效 |
| 示例与规则矛盾 | 模型跟示例走，不跟规则走 | 示例的权重高于抽象规则 |
| 使用"绝对禁止" | 模型在你禁止的地方出错 | 同第一点，负面强化 |
| 中英文混用 | 模型输出语言不稳定 | 英文术语在中文上下文中显得突兀 |
