# Argus 智能化分级体系 v2：计划-执行-审核-记录闭环

> **核心目标**：
> 1. 项目开发可控、高效、减少返工
> 2. 杜绝 AI 记忆有限导致的"改这个影响那个"
> 3. 所有工作有计划、有实施、有记录、有审核
>
> **设计原则**：
> - 文档即边界（Guardrails）— AI 的每一步操作都有文档依据
> - 文档是活的 — 所有角色按权限修改，不是锁死的
> - Prompt 不膨胀 — 规则外放到 Skills，操作封装为工具函数
> - 变更可追溯 — 每次改动都记录谁、为什么、改了什么、影响了什么

---

## 一、项目重量级（自动分级）

系统根据任务特征自动判断重量级，用户也可手动覆盖（`/level heavy`）。

| 重量级 | 典型特征 | 流程 | 文档启用 | 审核强度 | PM 总结 |
|--------|----------|------|----------|----------|---------|
| **Featherweight** | 单文件、<100 行、无依赖 | PM 直接执行 (`pmDirectExecute`) | 无 | 无 | 无需 |
| **Lightweight** | 2-5 文件、<500 行、单一功能 | PM → SE → AP 快速验证 | L0 概要 | PM 验收 + AP 轻量 | 可选 |
| **Medium（基线）** | 多模块、<5000 行、有内部依赖 | PM + SE + AP（完整三角色） | L0 + L1 + L2 | AP 强审（编译/测试/文档一致性） | 必须 |
| **Heavy** | 大型项目、多团队协作、长周期 | 全套 + K Tool 影响分析 | L0 + L1 + L2 + L3 全量 | AP 架构审核 + 影响分析 + 横向审核 | 必须 |

## 二、Skills & Rules 体系（Prompt 防膨胀）

### 为什么需要

竞品（Cursor / Windsurf / Claude Code）都用外部文件管理规则，避免 Prompt 膨胀：

| 产品 | 存储路径 | 用途 |
|------|----------|------|
| Cursor | `.cursor/rules/*.md` | 项目规则 |
| Windsurf | `.windsurf/rules/*.md` | 团队共享规则 |
| Claude Code | `.claude/skills/` | 三层技能包 |

Argus 采用同样的思路：**Prompt 只写"你是谁"，Rules 写"你怎么做"**。

### 目录结构

```
.argus/
├── skills/                          # Skills（规则文件，AI 启动时加载）
│   ├── _system/
│   │   ├── workflow.md              # 核心流程规则（PM→SE→AP 闭环）
│   │   ├── permissions.md           # 权限模型（谁能改什么）
│   │   └── review-checklist.md      # AP 审核清单
│   ├── project-level.md             # 分级标准（从智能化分级提取）
│   └── coding-standards.md          # 编码规范（可选，用户自定义）
│
├── docs/                            # 活文档（AI 可读写，受权限约束）
│   ├── PROJECT_PLAN.md              # 当前任务计划（活的，随进度更新）
│   ├── ARCHITECTURE.md              # L0: 全局架构
│   ├── modules/                     # L1: 模块文档
│   │   ├── core.md
│   │   ├── ai.md
│   │   └── chat.md
│   ├── submodules/                  # L2: 子模块/包文档
│   │   └── debugger/
│   │       └── client.go.md
│   └── functions/                   # L3: 函数/类型文档
│       └── se_prompt/
│           └── ProcessTaskWithTools.md
│
├── logs/                            # 变更记录（只追加，不可改）
│   └── CHANGELOG.md                 # 全局变更审计链
│
└── summaries/                       # 任务总结归档
    └── 2026-06-09-hello-world.md    # 每次完成的 SUMMARY
```

### 加载机制

```
AI 启动时（每个会话开始）：
  1. 读取 .argus/skills/_system/*.md     → 注入 System Prompt 前部
  2. 读取 .argus/skills/*.md              → 注入 System Prompt 中部
  3. 读取 .argus/docs/PROJECT_PLAN.md     → 注入当前任务上下文
  4. 读取当前任务相关的 L0-L3 文档         → 注入相关模块知识

结果：Prompt 本身永远 ~50 行，所有规则来自磁盘文件
```

## 三、权限模型（术业专攻）

所有角色都能看到所有文档，但**只能修改自己职责范围内的**。

### 权限矩阵

| 操作 | PM | SE | AP | 用户 |
|------|:--:|:--:|:--:|:----:|
| **读** 所有文档 | ✅ | ✅ | ✅ | ✅ |
| **写** L0 架构文档 | ✅ | ❌ | ❌ | ✅ |
| **写** L1 模块文档 | ✅ | ❌ | ❌ | ✅ |
| **写** L2 子模块文档 | ❌ | ✅ | ❌ | ✅ |
| **写** L3 函数文档 | ❌ | ✅ | ❌ | ✅ |
| **写** PROJECT_PLAN.md | ✅(拆解) / SE(更新状态) | (仅checkbox) | ❌ | ✅ |
| **写** 代码文件 | ❌ | ✅ | ❌ | ✅ |
| **审核/驳回** | (返工指令) | ❌ | ✅ | ✅ |
| **生成 SUMMARY** | ✅ | ❌ | ❌ | — |

### 硬约束实现

权限不只靠 Prompt 约束，**工具函数层面强制校验**：

```go
// 工具函数内部做权限检查，不依赖 AI 自觉
func updateDoc(level string, path string, content string, role string) error {
    // 权限矩阵
    allowed := map[string]map[string]bool{
        "pm":  {"L0": true, "L1": true, "plan": true},
        "se":  {"L2": true, "L3": true, "plan_status": true}, // SE 只能更新 checkbox
        "ap":  {}, // AP 不能写任何东西
    }

    if !allowed[role][level] {
        return fmt.Errorf("权限拒绝: %s 无法修改 %s 级别文档", role, level)
    }
    // ... 执行写入
}
```

即使 AI 被 injection 或"忘记"了权限规则，代码层面也会拦截越权操作。

## 四、文档分级体系（L0～L3）

### 各层定义

| 层级 | 文档模板 | 内容 | 维护者 | 更新时机 |
|------|----------|------|--------|----------|
| **L0** | `ARCHITECTURE.md` | 项目整体结构、模块划分、技术选型、非功能需求 | PM | 架构变更、里程碑完成 |
| **L1** | `modules/<name>.md` | 模块职责、对外接口、依赖关系、配置说明 | PM | 模块接口变更、新增模块 |
| **L2** | `submodules/<path>.md` | 包/子模块设计、关键类型、导出函数概要 | SE | 代码变更后同步更新 |
| **L3** | `functions/<symbol>.md` | 单个函数签名、行为说明、参数/返回值、示例 | SE | 函数修改后同步更新 |

### 启用规则

- Featherweight：不启用任何文档
- Lightweight：仅 L0（自动生成的概要）
- Medium：L0 + L1 + L2
- Heavy：全量 L0 + L1 + L2 + L3

### 文档与代码同步原则

```
核心原则：代码和文档必须保持一致，不一致 = 缺陷

SE 改代码的完整动作序列：
  1. 读相关 L2/L3 文档 → 了解当前定义
  2. 写/改代码 → 实现功能
  3. 更新 L2/L3 文档 → 同步变更（调用 update_doc 工具）
  4. 记录到 CHANGELOG.md → 追溯依据（调用 log_change 工具）
  5. 调用 complete_task → 标记完成

PM 改架构的完整动作序列：
  1. 分析影响范围（哪些 L1 模块受影响）
  2. 更新 L0 ARCHITECTURE.md
  3. 更新受影响的 L1 模块文档
  4. 记录到 CHANGELOG.md
  5. 通知 SE 按新架构调整
```

## 五、AI 记忆架构（解决深度 + 广度双重局限）

### 问题本质

AI 的记忆有三个维度，每个都有硬限制：

```
┌─────────────────────────────────────────────────────┐
│                  AI 记忆三维度                        │
│                                                     │
│  时间维度（广度-时间）                                │
│  ├── 问题：Session A 改了 X，Session B 忘了 X        │
│  ├── 表现：重复劳动、冲突修改、回归缺陷               │
│  └── 解决：CHANGELOG.md（变更时间线）                 │
│                                                     │
│  空间维度（广度-范围）                                │
│  ├── 问题：项目有 20 个模块，AI 只能看到当前任务的 3 个 │
│  ├── 表现：改 A 破坏 B 的接口、遗漏依赖更新           │
│  └── 解决：L0→L1→L2→L3 分层文档 + 按需加载          │
│                                                     │
│  细节维度（深度）                                     │
│  ├── 问题：写第 300 行时忘了第 50 行做了什么          │
│  ├── 表现：函数签名不一致、变量名冲突、逻辑断裂        │
│  └── 解决：L3 函数级文档 + 工具函数封装               │
│                                                     │
│  Context Window（物理上限）                           │
│  ├── 问题：以上所有东西加起来超过 token 上限           │
│  └── 解决：智能加载策略（见下方）                      │
└─────────────────────────────────────────────────────┘
```

### 5.1 时间记忆：CHANGELOG.md

**解决什么**：跨 Session 失忆

```
.argus/logs/CHANGELOG.md（只追加，永不删除）

## [2026-06-09 14:30] @PM — 架构调整
- **任务**: #T003 新增 Debugger 模块
- **变更**: 新增 internal/debugger/ 模块
- **影响**: L0 ARCHITECTURE.md 更新, 新增 L1 debugger.md
- **关联计划项**: PROJECT_PLAN.md #3

## [2026-06-09 14:45] @SE — 实现 DAP Client
- **任务**: #T003.1 编写 debugger/client.go
- **变更**: 新增 client.go (~300行), 更新 L2 debugger/client.go.md, 新增 L3 ProcessTaskWithTools.md
- **影响**: 无（新增文件）
- **关联计划项**: PROJECT_PLAN.md #3.1

## [2026-06-09 15:10] @AP — 审核通过
- **审核对象**: #T003.1
- **结论**: PASS — 代码与 L2/L3 文档一致, 无越权操作
- **关联计划项**: PROJECT_PLAN.md #3.1 → [x]
```

### 5.2 空间记忆：L0→L1→L2→L3 分层文档

**解决什么**：全局视野受限，不知道改这个会影响那个

| 层级 | 覆盖范围 | 加载时机 | 解决的问题 |
|------|----------|----------|-----------|
| **L0** | 整个项目架构 | 始终加载 | "这个项目长什么样？模块怎么划分的？" |
| **L1** | 当前模块及相邻模块 | 任务开始时加载 | "我改的这个模块对外暴露了什么接口？谁依赖它？" |
| **L2** | 当前子模块/包 | SE 执行时加载 | "这个包里有哪些类型和函数？" |
| **L3** | 当前正在编辑的函数 | 按需加载 | "这个函数签名是什么？参数含义？" |

**关键设计：不是一次性全加载，而是按需加载**

```
SE 开始执行任务 #T003.1 "编写 debugger/client.go"：

Step 1: 加载 L0（~200 tokens）→ 知道 debugger 是独立模块
Step 2: 加载 L1 debugger.md（~500 tokens）→ 知道模块接口定义
Step 3: 加载 L2 debugger/client.go.md（~800 tokens）→ 知道包内结构
Step 4: 开始写代码... 遇到 ProcessDAPMessage 函数
Step 5: 加载 L3 ProcessDAPMessage.md（~300 tokens）→ 知道函数细节

总消耗: ~1800 tokens（而非整个项目 ~50000 tokens）
如果 Context Window 不够，优先丢弃 L3（可以重新加载），保留 L0+L1
```

### 5.3 细节记忆：工具函数封装 + L3 文档

**解决什么**：写长代码时忘记前面的细节

| 场景 | 没有 L3 / 工具函数 | 有 L3 / 工具函数 |
|------|-------------------|-----------------|
| 写第 300 行时忘了第 50 行的类型定义 | AI 瞎猜或重新生成不一致的类型 | 读 L3 文档获取精确签名 |
| 更新 5 个相关函数后漏了一个 | AP 审核才发现 | `log_change` 强制记录每个变更 |
| 函数参数改了但调用方没同步 | 运行时报错 | L3 文档记录调用关系，AP 审核检查 |

**核心思想：AI 不需要记住所有细节，只需要知道去哪里查。**

```
传统方式（靠 AI 记忆）：
  AI context 里塞满代码 → token 爆炸 → 截断 → 丢失关键信息 → 出错

Argus 方式（文档即外挂记忆）：
  AI context 里只有：
    ├── Skills/Rules（怎么做）        ~1000 tokens
    ├── 当前任务描述                  ~500 tokens
    ├── L0 + L1（全局概览）            ~700 tokens
    ├── L2（当前包）                  ~800 tokens
    └── 正在写的 1-2 个 L3            ~600 tokens
    总计: ~3600 tokens ← 远低于上限
  
  需要更多细节？→ 调用 read_file 或 read_doc 工具按需读取
  写完一个函数？→ 调用 update_doc(L3) 保存细节，context 可以释放
```

### 5.4 Context Window 智能管理（防溢出）

当项目变大、文档变多，Context Window 可能不够。需要加载策略：

```go
// 伪代码：智能上下文加载器
func LoadContextForTask(task Task, role string, budget int) []string {
    var items []string
    used := 0

    // P0: 永远加载（不可裁剪）
    items = append(items, loadSkills())       // 规则
    used += estimateTokens(loadSkills())
    items = append(items, loadPlanSummary())  // 计划摘要（非全文）
    used += estimateTokens(loadPlanSummary())

    // P1: 高优先级（尽量保留）
    if role == "pm" {
        items = append(items, loadL0()) // PM 需要 L0 全局视野
        used += estimateTokens(loadL0())
    }
    if role == "se" {
        items = append(items, loadL1ForModule(task.Module)) // SE 只看自己模块的 L1
        items = append(items, loadL2ForSubmodule(task.Submodule))
        used += ...
    }

    // P2: 按需加载（可随时释放）
    if budget - used > 0 {
        items = append(items, loadRecentChangelog(5)) // 最近 5 条变更
    }

    // P3: 低优先级（最先被裁剪）
    if budget - used > 0 {
        items = append(items, loadL3ForCurrentFunction(task.Function))
    }

    return items
}
```

**裁剪优先级**（从低到高，Context 不够时先丢这些）：

| 优先级 | 内容 | 裁剪后果 | 补救措施 |
|--------|------|----------|----------|
| P3 | L3 函数文档 | 不知道某个函数细节 | 调用 read_doc 工具按需读 |
| P2 | CHANGELOG 历史 | 不知道最近改过什么 | 只保留最近 5 条 |
| P1 | L1 相邻模块 | 不知道谁依赖我 | K Tool 影响分析补 |
| **P0** | **Skills + Plan 摘要** | **不可裁剪** | — |

### 5.5 Session 恢复机制（完整版）

```
AI 新会话启动时的完整恢复流程：

  ① Load Skills/Rules          → 我是谁、我能做什么、规则是什么
  ② Load PROJECT_PLAN.md       → 当前在做什么、做到哪了
  ③ Load 最近 N 条 CHANGELOG    → 最近发生了什么（N=默认10条）
  ④ Load L0 ARCHITECTURE.md     → 项目全局结构
  ⑤ 根据 current_task 判断角色:
      ├── 如果是 PM → 加载相关 L1 模块文档
      ├── 如果是 SE → 加载当前任务的 L1+L2+L3
      └── 如果是 AP → 加载待审核项涉及的 L0-L3 + diff
  ⑥ Token 预算检查:
      ├── 如果超预算 → 按 P3→P2→P1 顺序裁剪
      └── 裁剪的内容标记为 [lazy-load]，需要时再读

结果：AI 在 3 秒内恢复到"知道一切必要信息"的状态
```

## 六、AP 审核清单（具体化）

AP 不是模糊地"检查一致性"，而是逐项打勾：

```markdown
## AP 审核清单

### 必须全部通过才可 APPROVE

#### A. 代码质量
- [ ] 编译通过（go build / npm build 等）
- [ ] 测试通过（如有测试）
- [ ] 无明显安全漏洞

#### B. 文档一致性（核心！）
- [ ] SE 修改的每个代码文件，都有对应的 L2/L3 文档更新
- [ ] L2/L3 文档描述的行为与实际代码一致
- [ ] SE 没有越权修改 L0 或 L1 文档
- [ ] 新增文件已在对应层级的文档中注册

#### C. 计划对齐
- [ ] 所有代码变更都能追溯到 PROJECT_PLAN.md 中的某个计划项
- [ ] 没有超出任务范围的变更（scope creep）
- [ ] 计划文档中对应项的 checkbox 已被 SE 标记

#### D. 变更追踪
- [ ] CHANGELOG.md 中有本次变更的记录
- [ ] 记录包含：谁、做了什么、影响了哪些文件/文档、关联哪个计划项

### 驳回时必须给出原因
@SE 请返工：
- 驳回项：B.2 / C.1
- 原因：client.go 第45行新增了 DebugStart 函数，但 L3 文档未更新
- 要求：补充 L3 functions/se_prompt/DebugStart.md
{"review_result":"reject","reason":"B.2: L3文档缺失","items":["B.2","C.1"]}
```

## 七、C 监控增强（一等公民）

C 不再只是超时监控，而是全程参与质量保障：

| 监控项 | 触发条件 | 动作 |
|--------|----------|------|
| 任务超期 | PROJECT_PLAN.md 中 `[ ]` 项超过 due 日期 | 提醒 PM |
| 文档漂移 | 代码变更后 30 秒内没有对应的 L2/L3 更新 | 提醒 SE |
| 循环检测 | 同一任务 @SE 超过 3 次 | 强制转 AP 审核 |
| 越权尝试 | 工具函数返回权限拒绝错误 | 记录到 CHANGELOG 并告警 |
| 长时间无进展 | 单个任务执行超过 10 分钟无输出 | 提醒用户 |

## 九、分阶段实施

### 阶段 1：基础闭环（MVP）

**目标**：计划可追踪、变更可记录、审核有依据

| 序号 | 工作 | 产出 |
|------|------|------|
| 1 | 创建 `.argus/skills/` 目录 + `_system/` 下 3 个 skill 文件 | workflow.md / permissions.md / review-checklist.md |
| 2 | 实现 4 个工具函数 | `update_plan_doc` / `update_doc` / `log_change` / `generate_summary` |
| 3 | 创建 `PROJECT_PLAN.md` 模板 + 自动生成逻辑 | PM 拆解时自动写入 |
| 4 | 创建 `CHANGELOG.md` + 追加逻辑 | 每次操作自动记录 |
| 5 | 修改 PM/SE/AP 提示词 | 从 ~200 行精简到 ~50 行，规则移入 skills |
| 6 | PM 总结生成（Medium/Heavy 级别） | AP 通过后触发 |

**不涉及**：L0~L3 分级文档（阶段 2）、K Tool 影响分析（阶段 3）、前端面板（阶段 4）

### 阶段 2：文档分级 + 权限 enforcement

| 序号 | 工作 | 产出 |
|------|------|------|
| 1 | 创建 `.argus/docs/` 目录结构（L0~L3） | 含 YAML frontmatter 模板 |
| 2 | `update_doc()` 加入权限校验 | SE 改 L0/L1 时返回权限拒绝 |
| 3 | SE 提示词增加"改代码必须同步更新 L2/L3" | 作为 skill 规则而非 prompt |
| 4 | AP 审核加入"文档一致性检查"步骤 | 使用第六节的审核清单 |

### 阶段 3：重型增强

| 序号 | 工作 | 产出 |
|------|------|------|
| 1 | K Tool 基础版（依赖分析） | `GetImpactChain` / `GetUpstreamDependents` |
| 2 | Heavy 级别自动启用 L3 + 影响分析 | AP 审核时调用 K Tool |
| 3 | `determineProjectLevel()` 自动分级 | 基于文件数/行数/依赖数 |

### 阶段 4：前端展示

| 序号 | 工作 | 产出 |
|------|------|------|
| 1 | 项目信息面板 | 显示级别、计划进度、文档统计 |
| 2 | CHANGELOG 查看器 | 时间线展示所有变更 |
| 3 | 文档树浏览器 | L0→L1→L2→L3 展开查看 |

## 十、验收标准

### 阶段 1 验收

- [ ] 用户说"创建 hello.go"，PM 自动生成 PROJECT_PLAN.md 并拆解任务
- [ ] SE 完成后自动标记 checkbox `[x]`
- [ ] CHANGELOG.md 有完整的变更记录（谁、改了什么、关联哪个计划项）
- [ ] AP 审核使用清单逐项检查，不一致则驳回
- [ ] AP 通过后 PM 自动生成 SUMMARY.md 并归档
- [ ] Prompt 体积不增反减（规则移入 skills 后）
- [ ] 新 session 启动时，AI 能从 CHANGELOG 恢复上下文

### 最终目标

> 一个 Argus 管理的项目，即使换了一个全新的 AI session，
> 也能通过 PROJECT_PLAN.md + CHANGELOG.md + L0~L3 文档，
> 瞬间理解项目全貌、知道做到哪了、下一步该做什么。
>
> 这就是"杜绝 AI 记忆有限"的终极方案——**文档即记忆**。
