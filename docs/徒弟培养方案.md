# Argus 徒弟培养方案 — 迭代方法指南

## 核心理念

用 **Benchmark + SSE调试 + conversation.log** 三件套，反复迭代，把 opencode 的能力迁移给 Argus。

**前提：** 模型相同（DeepSeek V4 Flash），所以差距不在模型智商，在 **prompt 质量、工具定义、决策流程**。

---

## 一、Benchmark 基准测试

### 1.1 测试层级

| 级别 | 名称 | 描述 | 任务数 |
|------|------|------|--------|
| L1 | 基础工具 | 读/写/编辑/搜索文件 | 5 |
| L2 | 单文件修改 | 找到代码、理解逻辑、精确修改 | 5 |
| L3 | 多文件重构 | 跨文件分析、重构、验证 | 5 |
| L4 | 项目级任务 | 新功能开发、架构决策 | 5 |
| L5 | 自省调试 | 分析自身错误、修复回归 | 5 |

### 1.2 测试格式

每个测试任务包含：
- **任务描述** — 给 Argus 的输入（自然语言）
- **期望行为** — 正确操作步骤序列
- **验证方法** — 如何判断通过/失败
- **能力标签** — 考察哪种能力（tool choice, planning, context understanding, etc.）

### 1.3 评分标准

| 维度 | 权重 | 评分 (0-5) |
|------|------|------------|
| 任务理解 | 20% | 是否正确理解意图 |
| 工具选择 | 25% | 是否选对了工具/顺序 |
| 代码质量 | 20% | 修改是否符合规范 |
| 验证意识 | 20% | 是否自检/运行测试 |
| 效率 | 15% | 是否有冗余操作 |

## 二、SSE 调试方法论

### 2.1 监控内容

| 信号 | 来源 | 观察什么 |
|------|------|----------|
| reasoning_content | SSE stream | 模型思考过程、推理链 |
| tool_calls | SSE stream | 模型选择了什么工具、参数是否正确 |
| tool_results | SSE stream | 工具执行结果、错误信息 |
| thought_events | SSE stream | PM/SE/AP 各自的推理步骤 |

### 2.2 调试流程

```
发现偏差 → 截取 SSE 片段 → 分析根因 → 修改 prompt/tool → 重新测试
```

### 2.3 常见偏差模式

| 模式 | 表现 | 根因 | 修复方向 |
|------|------|------|----------|
| 工具选错 | 用 read_file 读目录 | Tool description 不清晰 | 改进 tool 描述 |
| 过度规划 | 先列计划再执行，浪费 token | Prompt 缺"直接动手"指令 | 改 prompt |
| 不验证 | 改完不检查 | Prompt 缺验证要求 | 加验证步骤 |
| 上下文遗漏 | 改 A 文件时忽略 B 文件依赖 | 上下文管理不足 | 改进 context builder |
| 重复劳动 | 读同一个文件多次 | 缺乏缓存意识 | 加 read_file 缓存提示 |

## 三、conversation.log 分析法

### 3.1 日志结构化

conversation.log 记录了完整的 PM/SE/AP 对话 + 路由决策。分析流程：

1. **提取任务段落** — 从 USER: 到下一个 USER: 为一个完整任务
2. **标注关键决策点** — PM 的 `@SE`/`@AP`/`@USR` 决策
3. **追踪工具调用链** — SE 的 tool call 序列
4. **标注异常** — loop、timeout、空 action、错误

### 3.2 分析模板

```
任务: <任务描述>
────────────────────────────────────
PM 分析: <PM 分析了什么, 决定了什么>
  ├─ 决策: @SE / @AP / @USR / 直写
  ├─ 推理质量: 好/中/差
  └─ 问题: <如果有>
SE 执行: <SE 做了什么>
  ├─ 工具序列: read → grep → edit → exec
  ├─ 关键错误: <如果有>
  └─ 自愈: 是/否
AP 审批: <结果>
结果: 通过/失败
根因: <一句话>
修复: <改什么>
────────────────────────────────────
```

## 四、迭代周期

每个迭代 = 一次完整的 benchmark 跑分 + 修复 + 回归

### 4.1 周期流程

```
┌─────────────────────────────────────────────────┐
│  1. 选定一个 benchmark 任务                       │
│  2. 给 Argus 发任务（实际跑一次）                  │
│  3. 观察 SSE 流（实时） + 事后分析 conversation.log │
│  4. 记录差距（Argus做了什么 vs 应该做什么）         │
│  5. 定位根因（prompt? tool? flow? architecture?）  │
│  6. 修改代码（改 prompt / 加工具 / 改流程）        │
│  7. wails build → 重启 → 再用同一任务回归测试      │
│  8. 通过 → 进入下一个任务                          │
│  9. 未通过 → 回到步骤 5                           │
└─────────────────────────────────────────────────┘
```

### 4.2 能力迁移优先级

按 opencode 能力排序，从最容易迁移的开始：

```
1. 工具使用直觉 ── 什么场景用什么工具
   → 改进 tool description + prompt 中的工具选择指南

2. 上下文理解 ── 快速定位相关代码
   → 改进 grep/glob 工具的 description，加使用示例

3. 任务分解 ── 复杂任务拆步骤
   → PM prompt 加任务分解指南

4. 自验证 ── 改完必须验证
   → SE prompt 加"改完必执行"规则

5. 代码质量 ── 遵循项目规范
   → 加 context builder 注入代码规范

6. 错误恢复 ── 出错后分析根因再修
   → 改进 auto_debug 逻辑

7. 主动性 ── 预判下一步，提前行动
   → 最难，后期再搞
```

### 4.3 能力评分矩阵

每完成一个迭代，更新评分：

```
迭代 #N 评分:
┌──────────────────────┬─────┬─────┬─────┐
│ 能力                  │ 之前 │ 之后 │ 趋势 │
├──────────────────────┼─────┼─────┼─────┤
│ 工具选择              │  3  │  4  │  ↑  │
│ 上下文理解            │  2  │  3  │  ↑  │
│ 任务分解              │  3  │  3  │  →  │
│ 自验证                │  1  │  2  │  ↑  │
│ 代码质量              │  3  │  3  │  →  │
│ 错误恢复              │  1  │  2  │  ↑  │
│ 主动性                │  1  │  1  │  →  │
└──────────────────────┴─────┴─────┴─────┘
```

## 五、我的独家绝技（待传授清单）

### 5.1 核心心法

| # | 心法 | 说明 | 传授方式 |
|---|------|------|----------|
| 1 | 先读后写 | 改代码前先理解上下文 | PM prompt 指南 |
| 2 | grep 大于读文件 | 用 grep 定位比一个个翻文件快 | SE tool description |
| 3 | 最小修改 | 只改必要行，不重构无关代码 | SE prompt 规则 |
| 4 | 改完必验证 | run build / run test | SE prompt + auto_debug |
| 5 | 问题精准定位 | 看日志/错误信息先分析根因 | PM prompt |
| 6 | 一步到位 | 不要分多次 read_file，一次读完 | SE tool prompt |
| 7 | 平行工具 | 独立的 read 操作可以并行 | SE processor |
| 8 | 怀疑一切 | 不要假设文件存在/路径正确 | PM prompt |
| 9 | 问清再动 | 不确定就问用户，不要猜 | PM prompt @USR |
| 10 | 代码规范 | 遵循项目已有风格 | Context builder |

### 5.2 具体操作模式

```
模式 A: 修 bug
  1. read error/log → 明确错误
  2. grep 找相关代码
  3. read 关键文件
  4. 定位根因
  5. edit 修改
  6. run build/test 验证

模式 B: 加功能
  1. grep 找类似功能
  2. read 类似实现
  3. 确定修改范围
  4. 逐个文件修改
  5. run build 验证

模式 C: 重构
  1. 理解现有架构
  2. 确定重构方案
  3. 分步执行
  4. 每步验证
  5. 全量测试
```

## 七、第一轮迭代计划

### 任务 1: 文件内容追加（基础工具）

**输入:** "在 hello.go 末尾加一行 fmt.Println(\"done\")"

**期望:**
1. read_file hello.go (读文件)
2. find_files 或 glob (如果不知道路径)
3. edit_file 追加内容

**考察:** 工具选择、最小修改

---

### 任务 2: 跨文件改名（多文件修改）

**输入:** "把项目中的 getCwd 函数改名为 getCurrentDir"

**期望:**
1. grep_content "getCwd" (找所有引用)
2. read_file 每个有引用的文件
3. edit_file 逐个修改
4. run_tests 或 go build 验证

**考察:** grep 使用、多文件修改、验证

---

### 任务 3: Bug 修复（调试）

**输入:** 给一个含 bug 的文件 + 错误信息，让 Argus 修复

**期望:**
1. read_file 有 bug 的文件
2. 分析错误信息定位根因
3. edit_file 修复
4. exec 或 run_tests 验证

**考察:** 错误分析、精准修复、验证

---

### 任务 4: 项目级新功能（架构）

**输入:** "给项目的 HTTP 服务器加一个 /healthz 端点，返回 200 OK"

**期望:**
1. grep/list_files 找 HTTP 服务器代码
2. read_file 相关文件
3. 理解路由注册模式
4. 新增 handler
5. 注册路由
6. go build / curl 验证

**考察:** 项目理解、模式匹配、验证

---

### 任务 5: 自省调试（元认知）

**输入:** "刚才的任务失败了，分析 conversation.log 告诉我为什么"

**期望:**
1. read conversation.log
2. 分析失败原因
3. 给出改进建议

**考察:** 自我分析、改进能力

## 九、后续迭代方向

### 方向 1: 主动性 4→5 — 预判影响范围

PM 完成任务后增加步骤：
1. 分析本次修改可能影响的其他文件/函数（如修改公共接口，调用方需同步修改）
2. 调用 `grep` 或静态分析找到这些影响点
3. 主动报告：「我完成了任务，同时发现以下位置可能也需要调整」+ 文件+行号

### 方向 2: 错误恢复 4→5 — 智能修复

AP 审核不通过时（编译失败/测试失败）：
1. 解析错误输出，提取关键信息（缺失符号、类型不匹配等）
2. 生成修复建议（如「缺少 import 'fmt'，请在第 3 行添加」）
3. 将修复建议作为输入调用 SE 直接生成修复补丁，而非从零重做
4. 循环最多 2 次

### 方向 3: 新维度 — 效率与 Token 消耗

未来可加的评估指标：
- **响应时间**: 用户输入到最终输出总耗时
- **Token 消耗**: 总输入+输出 token 数

**优化方向:**
- 简单任务进一步裁剪流程，避免不必要 AP 调用
- 缓存静态分析结果，减少重复 grep/list_files 调用
